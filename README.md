# 常に名前を付けて保存(Edge拡張機能) v1.3.0

ダウンロードするたびに、OSの「名前を付けて保存」ダイアログを**強制的に**開かせる拡張機能です。

## インストール方法(開発者モードで読み込み)

Microsoftストアには公開されていないため、「開発者モード」で手動読み込みします。

1. ダウンロードした `force-saveas-extension-v1.0.1.zip` を**展開(解凍)**する
   - 展開後、`manifest.json` が入っているフォルダを確認してください
   - ⚠️ Windowsの「すべて展開」で、zipと同じ名前のフォルダがさらに1つできて
     `manifest.json` がその中の、さらにもう1つ奥のフォルダに入ってしまう場合が
     あります。**`manifest.json` が直接見えている階層のフォルダ**を次の手順で選んでください。
2. Edgeで `edge://extensions` を開く
3. 左下(または左上)の **「開発者モード」** をオンにする
4. **「展開して読み込み」** をクリック
5. 手順1で確認した、`manifest.json` が直接入っているフォルダを選択する
6. 一覧に「常に名前を付けて保存」が追加されれば完了

## 使い方

- インストール直後は **有効(ON)** になっています
- ツールバーのアイコンをクリックすると **ON / OFF** を切り替えられます(バッジ表示で確認可能)
- OFFにすると、通常のダウンロード動作(Edgeの設定に従う)に戻ります

## 事前におすすめの設定

拡張機能とEdge本体の設定が競合して二重にダイアログが出るのを避けるため、
`edge://settings/downloads` の **「ダウンロードする前に各ファイルの保存先を確認する」は
オフ**にしておくことを推奨します。ダイアログの制御はこの拡張機能側だけで行います。

## 既知の制限事項

- **認証付き・期限付きURL**: ログインが必要なファイルや、一度きりの署名付きURL(S3の
  期限付きリンクなど)は、拡張機能がキャンセル後に同じURLで再取得しようとした際に
  失敗する場合があります。
- **blob: URL**: JavaScriptで生成された一時ファイル(`blob:`で始まるURL)は、
  ダウンロード元のページがURLをすぐに無効化(revoke)する実装の場合、再取得に
  失敗することがあります。
- **極端に小さい/速いファイル**: キャンセル処理が間に合う前にダウンロードが完了して
  しまう場合があります。その際は通常どおり(設定済みの保存先に)保存されます。
- 上記の失敗時はブラウザのコンソール(拡張機能の「service worker」の検証)に
  警告ログが出力されます。

## ファイル構成

- `manifest.json` — 拡張機能の定義(Manifest V3)
- `background.js` — ダウンロードを検知し、キャンセル→ saveAs 付きで再実行する本体処理
- `_locales/ja/messages.json` — 日本語UI文言(拡張機能名・説明文・ツールチップ等)
- `test-harness.js` — chrome.* APIをモックしたロジック検証用テスト(`node test-harness.js` で実行可能)

## 日本語化(i18n)について

`manifest.json` は `chrome.i18n` の正式な仕組み(`default_locale` + `_locales/ja/messages.json`)
を使って日本語化しています。`name` や `description` に直接文字列を書く代わりに
`__MSG_extName__` のようなプレースホルダを使い、実際の文言は
`_locales/ja/messages.json` 側で管理しています。

現在は日本語(`ja`)ロケールのみ用意しているため、ブラウザの表示言語設定に
かかわらず常に日本語で表示されます。もし英語版など他言語にも対応させたい場合は、
`_locales/en/messages.json` のように言語コードのフォルダを追加するだけで、
Edge側がブラウザの言語設定に応じて自動的に切り替えて表示してくれます。

## デバッグで見つかり、修正済みの不具合

000. **【重大】ブラウザを再起動すると過去のファイルが再びダウンロードされるバグ
    (v1.3.0で対策)**:
    ブラウザを終了して再起動すると、終了時点で完了していなかったダウンロードが
    再び動き出し、それが.tempファイル大量発生の原因になっていることが判明した。
    Chromiumはブラウザ終了時に進行中だったダウンロードの記録を保持しており、
    再起動時にそれらが動き出す(またはそう見える)ことがあると考えられるが、
    正直なところ、この拡張機能の外側にあたるブラウザ内部の厳密な挙動までは、
    今回の開発環境では実機のEdgeで動作確認ができないため完全には特定し
    きれていない。そのため、**原因を問わず症状を止めることを優先し**、
    以下2つの防御策を追加した:
    - **対策A**: ダウンロードの開始時刻(`startTime`)が現在時刻から
      10秒以上離れている場合、たった今の新規操作とは考えにくいため横取りしない
    - **対策B**: ブラウザ起動直後(`chrome.runtime.onStartup`発火後)5秒間は、
      セッション復元等が集中しやすいタイミングとして横取りを見合わせる
    もしv1.3.0を適用しても症状が再現する場合は、正確な原因特定のため
    「何を」「どんな操作で」ダウンロードした後に再起動したか、具体的な
    手順を教えていただけると助かります。
00. **【重大】保存ダイアログをキャンセルすると.tempファイルが残り、再ダウンロードが
    延々と始まるバグ(v1.2.0で修正)**:
    ユーザーが「名前を付けて保存」ダイアログで**キャンセル**を選ぶと、その
    ダウンロードは「中断(interrupted)」状態のままダウンロード履歴に残って
    しまっていた。この中断状態のまま残ったエントリが、後になって何らかの
    形で再度検知され、繰り返しダウンロードが始まる原因になっていた。
    → 自分がsaveAs付きで再発行したダウンロードのIDを記録しておき、
    それが**中断状態になった瞬間に、ダウンロード履歴から即座に消去する**
    ように修正。ダイアログでキャンセルした場合、その場できれいに片付く。
    ⚠️ 既にダウンロードフォルダに溜まってしまった一時ファイルは、
    この修正では自動的に削除されません。エクスプローラー等でダウンロード
    フォルダ内をご確認のうえ、手動で削除してください。
0. **【重大】ブラウザを閉じると大量の一時ファイルが延々と生成されるバグ(v1.1.1で修正)**:
   Edgeには既定で「終了してもバックグラウンドで拡張機能の実行を続行する」設定が
   オンになっている。この状態でウィンドウをすべて閉じても拡張機能は動き続けるが、
   `saveAs:true` で「名前を付けて保存」ダイアログを開こうとしても、表示先の
   ウィンドウが存在しない。すると、ダイアログを出せないままファイルが仮の名前で
   自動保存され、それを拡張機能が「新しい元のダウンロード」だと誤認識して
   再度横取り→また自動保存…を延々と繰り返す無限ループになっていた。
   → **開いているウィンドウが1つも無い場合は横取りを行わない**ように修正。
   さらに保険として、**同じURLへの横取りが短時間に3回を超えたら自動的に
   あきらめる安全装置**も追加(原因が完全には特定できない別のループ経路が
   将来見つかった場合への多重防御)。
   ⚠️ **この修正を入れても、既にダウンロードフォルダに溜まってしまった
   大量の一時ファイルは自動的には削除されません。** お手数ですが、
   エクスプローラー等でダウンロードフォルダ内の見覚えのない `.tmp` /
   `.crdownload` 等のファイルをご自身で確認・削除してください。
   また、念のため `edge://settings/system` の
   「Microsoft Edge が終了してもバックグラウンドの拡張機能およびアプリの
   実行を続行する」をオフにしておくことも推奨します。
1. **ダウンロード消失バグ**: ダウンロード履歴からの削除(`erase`)が何らかの理由で
   失敗すると、再ダウンロード処理そのものが止まってしまい、ファイルが
   キャンセルされたまま何も保存されなくなっていた。→ `cancel` と `erase` の
   エラー処理を分離し、`erase` の失敗は無視して再ダウンロードを続行するよう修正。
2. **service worker再起動への耐性不足**: 「自分自身が再発行したダウンロードか」を
   判定するための管理状態を単純なメモリ上の変数(Map)に持っていたため、
   Manifest V3のservice workerが処理の途中で再起動すると状態が失われる
   可能性があった。→ `chrome.storage.session` に保存するよう変更。
3. **同一URL同時ダウンロード時の競合状態**: 上記の状態管理が非同期の
   読み込み→書き込みだったため、同じURLへのダウンロードがほぼ同時に
   発生すると更新が競合し、余計なキャンセル/再ダウンロードが発生することが
   テストで確認された。→ 処理をキューで直列化するロックを追加して解消。
4. **過剰な権限要求**: `host_permissions: ["<all_urls>"]` は
   `chrome.downloads.download()` の実行に不要であり、インストール時に
   不必要に広い権限(「すべてのWebサイトのデータの読み取りと変更」)を
   要求していた。→ 削除。

いずれも `test-harness.js` の自動テストで再現・検証済みです。
